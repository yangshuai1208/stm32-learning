# Day14 学习笔记：FreeRTOS + DHT11 + OLED 温湿度显示

## 今天做了什么
今天在已经跑稳的 FreeRTOS 基础工程上，继续接入了 DHT11 温湿度传感器，并完成了 OLED 显示和串口输出，做出了一个基础的多任务温湿度监测系统。

## 今天完成的功能
1. 新增 `sensorTask`
2. 在 `sensorTask` 中读取 DHT11 温湿度数据
3. 使用共享结构体保存温湿度结果
4. 在 `ledTask` 中把温湿度数据显示到 OLED
5. 在 `uartTask` 中把温湿度数据显示到串口
6. 保留 LED 翻转作为系统心跳

## 今天的核心知识点

### 1. 任务职责拆分
今天项目里有三个主要任务：

- `sensorTask`：只负责采集
- `ledTask`：只负责显示
- `uartTask`：只负责输出和心跳

这样做的好处是：
- 结构清晰
- 模块职责明确
- 后续扩展更方便

### 2. 为什么先用共享结构体
今天一份温湿度数据需要同时被 OLED 和串口两个任务使用。  
这个场景下，先用共享结构体最简单直观，便于验证整体功能。

### 3. DHT11 的读取特点
DHT11 对时序要求比较高，读取过程本质上是忙等待加延时判断。  
所以：
- 必须先初始化 DWT
- 必须先初始化 DHT11 数据引脚
- 不适合放在中断里做

### 4. OLED 显示先用英文
今天 OLED 没有先上中文温湿度，而是直接显示英文：
- `DHT11`
- `TEMP`
- `HUMI`

这样做的优点是：
- 简单
- 直观
- 方便先把链路跑通

### 5. 错误状态也要显示
如果 DHT11 读取失败，不应该什么都不显示。  
今天的处理方式是：
- OLED 显示 `READ ERROR`
- 串口输出 `DHT11 READ ERROR`

这样更容易调试。

## 今日实验现象
### 正常情况下
- 串口输出温湿度数据
- OLED 显示温湿度
- 红灯周期翻转

### 异常情况下
- 串口输出 `DHT11 READ ERROR`
- OLED 显示 `READ ERROR`

## 今天遇到的问题
1. 编译时 `DHT11_DATA_TYPEDEF` 未定义
2. `sprintf` 和 `DHT11_ReadData` 未声明
3. 没有实验现象时，难以判断到底是任务没跑、OLED 没初始化，还是 DHT11 没读到
4. 任务多了之后，排错比之前单任务复杂很多

## 今天是怎么解决的
- 给 `freertos.c` 补上 `bsp_dht11.h`
- 补上 `stdio.h` 和 `string.h`
- 梳理任务职责
- 明确实验现象
- 用串口、OLED、LED 三条链路共同判断系统状态

## 今日收获
今天最大的收获不是多接了一个传感器，而是开始真正体会到：

- 多任务系统里要先拆职责
- 采集、显示、输出要分开
- 调试时必须设计“错误信息反馈”
- 一个嵌入式项目开始有了系统感
