# Day16 学习笔记：状态监测与异常统计 v2

## 今天做了什么
今天没有继续加新模块，而是在已经完成的 FreeRTOS + DHT11 + OLED + 串口工程基础上，继续做状态管理和异常统计，让系统不只是能显示温湿度，还能告诉我“系统现在运行得怎么样”。

## 今天完成的主要内容
1. 删除无用的默认任务
2. 保留 3 个核心任务
3. 升级共享数据结构体
4. 增加成功采集次数统计
5. 增加失败次数统计
6. OLED 增加状态显示
7. 串口增加状态输出

## 今天最核心的变化

### 1. 共享数据结构体不再只存温湿度
之前更多只是存：

- 温度
- 湿度
- valid

今天进一步增加了：

- `ready`
- `update_count`
- `error_count`

这样一来，系统就能区分：

- 还没拿到第一组数据
- 当前这次读取失败
- 到底成功采集了多少次
- 到底失败了多少次

这比单纯显示一个温湿度值更像真正的项目。

### 2. 三任务分工更清楚
今天项目里的三个任务分别是：

#### sensorTask
只负责采集 DHT11 数据，并更新共享结构体。

#### ledTask
只负责 OLED 显示，不直接读 DHT11。

#### uartTask
只负责串口输出，不直接读 DHT11。

这种拆分方式比前面更清晰，也更方便后续扩展。

### 3. 引入 ready 的意义
今天新增的 `ready` 很重要。

因为系统刚启动时，还没有采到第一组数据。  
如果没有 `ready`，OLED 可能会直接显示无意义的默认值。  
现在用了 `ready` 以后，刚启动时可以显示：

- `WAIT DATA`

这样状态就更清楚了。

### 4. 引入 update_count 和 error_count 的意义
今天最大的收获之一就是意识到：

一个系统值不值钱，不只是看它“有没有数据”，还要看它“有没有状态反馈”。

#### `update_count`
表示成功采集了多少次。

#### `error_count`
表示失败了多少次。

这样做的好处：

- 便于做稳定性测试
- 便于判断系统是不是一直在更新
- 便于后续 README 和项目讲解

## 今天的实验现象

### 刚启动时
OLED 显示：

- `DHT11`
- `WAIT DATA`

说明：
- OLED 工作正常
- 任务已经开始跑
- 传感器数据还没准备好

### 采集成功时
OLED 显示：

- `DHT11`
- `TEMP:xx.xC`
- `HUMI:xx.x%`
- `CNT:xx`

串口输出：

- `DHT11 TEMP:xx.xC HUMI:xx.x% CNT:xx`

### 采集失败时
OLED 显示：

- `DHT11`
- `READ ERROR`
- `ERR:xx`

串口输出：

- `DHT11 READ ERROR CNT:xx ERR:xx`

## 今天学到的关键点

### 1. “能跑”和“可观测”不是一回事
以前更多是在追求：

- 能读到值
- 能显示出来

今天开始理解到：

- 采集是否成功
- 失败了几次
- 系统是不是还在更新
- 当前状态是什么

这些信息也很重要。

### 2. 共享数据不仅能存结果，还能存状态
今天最直观的收获就是：

共享结构体里不只是塞传感器数据，也可以放：
- 状态位
- 统计计数
- 系统运行信息

### 3. 嵌入式项目要学会“给自己留调试信息”
以后写代码不能只想着“显示最终结果”，还要想着：

- 如果失败了怎么办
- 我怎么知道它失败了
- 我怎么知道它是不是一直在跑

今天的 `ready / valid / update_count / error_count` 就是在做这件事。
