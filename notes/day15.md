# Day15 学习笔记：系统整合与稳定化 v1

## 今天做了什么
今天没有继续盲目加新模块，而是把前一天已经完成的 FreeRTOS + DHT11 + OLED + 串口工程做了一次整理和稳定化处理，让这个项目开始更像一个真正的小系统。

## 今天的核心目标
今天的核心不是“多学一个外设”，而是：

- 整理当前工程结构
- 明确任务职责
- 提高运行稳定性
- 为后续继续扩展做准备

## 今天保留的功能
当前系统仍然包含以下核心功能：

1. DHT11 温湿度采集
2. OLED 动态显示温湿度
3. 串口动态输出温湿度
4. LED 运行心跳
5. FreeRTOS 多任务调度

## 今天整理的重点

### 1. 主函数只负责初始化和启动系统
今天进一步明确了 `main.c` 的职责：

- 初始化外设
- 初始化传感器和 OLED
- 创建任务
- 启动 FreeRTOS 调度器

真正的业务逻辑不应该继续写在 `main()` 末尾的 `while(1)` 里。

### 2. 三任务职责进一步明确
今天重新梳理了任务分工：

#### sensorTask
- 只负责采集 DHT11
- 只负责更新共享数据

#### ledTask
- 只负责 OLED 显示
- 不直接访问 DHT11 采集逻辑

#### uartTask
- 只负责串口输出
- 同时负责 LED 心跳

这种分工方式让代码更清晰，也更符合后续项目扩展思路。

### 3. 今天开始关注“稳定运行”
前面更多是在关注“能不能跑起来”。  
今天开始关注的是：

- 能不能持续跑
- OLED 会不会花屏
- 串口会不会中断
- DHT11 会不会长时间异常
- 任务会不会互相影响

## 今天学到的关键点

### 1. 任务拆分不只是为了好看
把采集、显示、输出拆开以后，最大的好处是：

- 更容易定位问题
- 更方便修改
- 更适合后续扩展
- 面试时也更容易讲清楚

### 2. 共享数据结构体是当前阶段最合适的方案
目前一份温湿度数据要同时给 OLED 和串口使用。  
这个阶段使用共享结构体最直观，也最容易先把系统跑稳。

### 3. 嵌入式项目的价值不只是“功能多”
今天最大的感受是：

一个项目真正值钱的地方，不是模块接得多，而是：

- 结构清晰
- 逻辑清楚
- 能稳定运行
- 能写成 README 和项目说明

### 4. 稳定性测试很重要
今天开始有意识地去看：

- 串口会不会持续打印
- OLED 会不会一直正常刷新
- LED 心跳是不是持续存在
- DHT11 是偶发失败还是一直失败

这比只看“下载后亮一下”更有意义。

## 今天观察到的实验现象
### 正常情况下
- 串口输出启动信息
- OLED 显示温湿度
- 串口打印温湿度
- LED 周期翻转

### 异常情况下
- OLED 可能显示 `READ ERROR`
- 串口可能输出 `DHT11 READ ERROR`

这说明系统至少具备了“错误状态反馈”的能力。

## 今天发现的问题
1. `defaultTask` 其实没有实际作用
2. 任务命名还可以继续优化
3. 主函数里仍有部分前期调试残留
4. 显示任务的栈空间还需要继续留意
5. 目前只是基础整合版，还没做更长时间的稳定性测试

## 今天的收获
今天最大的收获不是增加了新模块，而是开始真正用“项目思维”看待这套工程：

- 什么应该放在初始化阶段
- 什么应该放在任务里
- 什么是业务逻辑
- 什么是调试残留
- 什么是系统级问题

## 明天可以继续做什么
下一步可以继续考虑：

1. 删除 `defaultTask`
2. 优化任务命名
3. 调整栈和任务优先级
4. 增加稳定性测试记录
5. 再决定是否继续做队列、中断或更多输入模块

## 今日总结
第15天的意义不在于再加多少功能，而在于把现有功能整理成一个更稳定、更清晰、更接近项目化的版本。  
这是从“做实验”往“做项目”过渡的一步。
