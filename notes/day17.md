# Day17 学习笔记：任务架构最终版

## 今天做了什么
今天没有继续加新外设，而是把前面已经完成的 FreeRTOS + DHT11 + OLED + 串口工程进一步整理，形成了一个更加清晰的三任务架构版本。

## 今天完成的核心内容
1. 保留 3 个核心任务
2. 明确三任务职责边界
3. 将显示任务正式独立出来
4. 统一 OLED 和串口的状态输出逻辑
5. 保留系统状态字段和统计字段

## 今天的三任务结构

### 1. sensorTask
负责：
- 调用 DHT11 驱动读取温湿度
- 更新共享数据结构体
- 成功时增加 `update_count`
- 失败时增加 `error_count`

理解：
这个任务是整个系统的数据源，只负责采集，不负责显示和输出。

### 2. displayTask
负责：
- 从共享结构体读取快照数据
- 在 OLED 上显示当前状态

显示规则：
- 没拿到数据：`WAIT DATA`
- 读取成功：显示温湿度和 `CNT`
- 读取失败：显示 `READ ERROR` 和 `ERR`

理解：
这个任务只负责显示，不直接访问 DHT11 驱动。

### 3. uartTask
负责：
- 从共享结构体读取快照数据
- 串口输出温湿度或错误状态
- 翻转 LED 做系统心跳

理解：
这个任务只负责外部输出和运行状态提示。

## 今天学到的关键点

### 1. 主函数和任务函数的职责要分开
今天进一步确认了：

- `main.c` 负责初始化和启动系统
- 真正业务逻辑应该放到任务里

这比把所有代码堆在 `while(1)` 里更清晰，也更符合 FreeRTOS 工程思路。

### 2. 三任务结构比前面更像真正项目
前几天更多是在做：
- 能不能采到数据
- 能不能显示
- 能不能打印

今天进一步变成：
- 谁负责采集
- 谁负责显示
- 谁负责输出

这就是“功能代码”和“架构代码”的区别。

### 3. 状态字段让系统更容易观察
今天这版继续保留了：

- `ready`
- `valid`
- `update_count`
- `error_count`

这样系统就能明确区分：

- 刚启动还没有数据
- 当前采集成功
- 当前采集失败
- 一共成功了多少次
- 一共失败了多少次

这让项目不只是“显示结果”，而是“能反馈自己运行状态”。

### 4. 快照思路很重要
显示任务和串口任务都先把全局共享数据复制到本地快照，再进行显示或输出。

这样做的好处是：

- 减少临界区占用时间
- 减少和采集任务的互相影响
- 数据逻辑更清楚

## 今天的实验现象

### 刚启动时
OLED 显示：
- `DHT11`
- `WAIT DATA`

说明系统已经起来，但还没拿到第一组数据。

### 正常运行时
OLED 显示：
- `DHT11`
- `TEMP:xx.xC`
- `HUMI:xx.x%`
- `CNT:xx`

串口输出：
- `DHT11 TEMP:xx.xC HUMI:xx.x% CNT:xx`

### 读取失败时
OLED 显示：
- `DHT11`
- `READ ERROR`
- `ERR:xx`

串口输出：
- `DHT11 READ ERROR CNT:xx ERR:xx`
